Watchlist

An app that let's you search for movies and save them to a watchlist.

  • HTML
  • CSS
  • JavaScript
  • OMDB API
Visit Site →

Read Project Details

This Watchlist app was built using HTML, CSS and Vanilla JavaScript. The mockup for the design was provided by Scrimba's Frontend Developer Career Path course curriculum and I chose to strictly adhere to the design specs to showcase my ability to build projects that meet a creative departments visual standards. A large part of the learning experience for this project was also to learn to use JavScripts Fetch API to fetch and return content from the Open Movie Database API.

Project Development Phases

First, I decided to tackle the apps general styles and structure using HTML and CSS.

Phase 1 Process Highlights:

  • I found a way to use CSS properties to center the headers background image in close proximity to the projects design using background-position
  • Learned how to apply a filter to the header background image to darken the overall color cast
  • Used the CSS vh property and flexbox to vertically center the film icon graphic in the center of the main div

The second phase was to use the JavaScript Fetch API to search for movie data provided by the Open Movie Database API (OMDB) and render that movie information into individually returned cards.

Phase 2 Process Highlights:

  • Read the OMDB documentation to learn how to use their API's query string parameters
  • I learned how to use JavaScript .value to inject the searched for value into the API fetch query string using template literals
  • Used the map method to render individual movie data returned by the search to individual movie cards.
  • Some values were missing from the objects key value pairs that were returned by some search returns. I needed to adjust the card styles using if statements to identify the edge cases. In the case of a missing Movie Cover artwork I created my own holder image.

The third phase was to use browser localstorage to remember movie data that had been pushed/saved to the "My Watchlist" page.

Phase 3 Process Highlights:

  • Understanding how localstorage works was a big hurdle for me to get over and it's one of the aspects of this projject that I was most proud of wrapping my mind around.
  • I learned how to set local storage when the add to watchlist button was clicked and then how to import and use that object data by utilizing getlocalstorage to list movies I had added to my watchlist.
  • Understanding how to use getlocalstorage in the movie search results page to use saved movie data to help target previously selected movies so that there was a way to remember and mark movies that had already been added to the watchlist.

The fourth and final phase in my process for this project was to figure out how to apply a "read more" button to the movie description data so that I could maintain a controlled snippet layout for the search results cards.

Phase 4 Process Highlights:

  • Maintaining the designs layout was important to me so I was determined to incorporate the "read more button" for each cards movie description
  • It was a challenge to figure out how to implement this feature of the design as my researh didn't uncover any information on how to implement this dynamically. I posted a request for help to the Scrimba coding forum and although the responses did send me in the right direction I was unable to use the information garnered there to come to a solution.
  • I used psuedo code to make the steps I wanted to try concrete in my mind and this approach ultimately let me to a solution for how to implement the read more button feature.

Lessons learned

The watchlist project taught me how valuable it is to take a step back and write out psuedo code to wrap my head around challenging coding problems. Doing this allowed me to break parts of the project down so that I could tackle smaller challenges individuallly to create solutions for more challenging functionalities. This was helpful to figure out how to apply a check mark for movies that had already been added to the watchlist as well as in figuring out how to get the read more button to work with dynamically generated movie descriptions.

A website that shares reviews for new indie, electronic, and jazz music releases

  • JavaScript

  • HTML
  • CSS

Read Project Details

Sweet Release is built using HTML, CSS, JavaScript, React and Firebase. The design for this site was conceptualized and designed by me. It was my first website to design and build using React. Music is a big part of my life and I love to discover talented new musical artists. Eventually I would like to grow Sweet Release into a full blown online resource for music reviews. Some of the more complex challenges related to building this project were learning how to use Firebase with React to build user authentication as well as learning how to push form data to Firebase.

Project Development Phases

The first phase for this project was deciding what kind of information I wanted to include for each album review and then building out a data.js file to hold that data.

Phase 1 Process Highlights:

  • I researched other music reviews websites to come up with a plan for what kind of data I wanted to share on the site.
  • I learned how to correctly format JSON data into an array so that it could be imported and accessed by the JavaScript map method in my App.jsx file
  • My album images weren't rendering so I did some research and discovered that I needed to import the image files into the data.js file so that React could access them.

The second phase was to use template literals and the JavaScript map method to output my data for each data item to the screen in my desired layout.

Phase 2 Process Highlights:

  • I learned how to use JavaScripts rest operator to pass my data through to my template component
  • When the data was passed through to my Card component I destructured each object item so that I could access the key value pairs more succinctly when designing my template.

The third phase was to design the branding, layout and UX and to choose color's for the site as well as to write the CSS.

Phase 3 Process Highlights:

  • I wanted to be professional in defining descriptive and easy to understand class names in my HTML template for my CSS.
  • One of my more challenging aspects of this project was learning how to open a modal for when people clicked the Read Review button and using useState for my first time in a self directed project to pass the correct album review data to the modal.
  • I learned how to import and use Font Awesome icons.
  • I did some design work to create a logo and website name that was inline with the content I was sharing.
  • I used @media to create a responsive design that looks great and behaves well on mobile, tablet and desktop devices.

The fourth phase in my process was to refactor my code to use React Router so that I could create login and admin pages for making it easier to enter album review data to the site.

Phase 4 Process Highlights:

  • I used Browser Router to create protected routes that redirect users to a login page if they try to access the admin component.
  • I set a up a Firebase database for holding server side authentication credentials to test against.

Quizzical App

A quiz app that displays a set of trivia questions for you to answer

  • React
  • JavaScript
  • HTML
  • CSS
  • OTDB API
Visit Site →

Read Project Details

Quizzical was built using HTML, CSS, Vanilla JavaScript and React in conjunction with the Open Trivia Database API (OTDB). I again, chose to stay true to the mock up provided by Scrimba with the exception of building the app to apply a responsive layout. One of the most challenging and enjoyable aspects for me while building this project was learning how to work with React's useEffect to handle API calls using fetch.

Project Development Phases

First, I again chose to work on the general styles and structure of the Quizzical app using HTML and CSS.

Phase 1 Process Highlights:

  • Working on the styles and structure of this app was fairly straight forward but one cool thing I learned was how to use background-position in CSS to position my graphic elements.
  • Adjusting the design for a mobile layout was fairly straight forward as well as I used a mobile first approach and @media to adjust the layout to meet the desktop mock-up specifications.

The second phase was to get a quick win by taking the user to the start quiz page when they clicked the start quiz button.

Phase 2 Process Highlights:

  • I worked with useState and a ternary operator to take the user to the quiz page after clicking the Start Quiz button or if the app had just loaded and the startQuiz state was set as false to present them with the initial landing page.

The third phase was to fetch the OTDB data inside a useEffect using the more standardized Async Await approach.

Phase 3 Process Highlights:

  • A big challenge I needed to overcome in this phase of the project was to make sure that my call to the OTDB API did not get stuck in a loop resulting in crashing the program. My solution was to create an if statement inside the useEffect that only initiated an API call when the starQuiz state had been set to true.
  • I worked to understand how to implement async await syntax to use a more modern approach to working with promises than I had in previous projects.
  • I learned how to use the try / catch syntax to make sure that if any errors were thrown I would recieve some clues as to what the error was by having the error logged to the console.

The fourth phase of the Quizzical project was to shuffle the output for the possible quiz answers making sure to include the correct answer and 3 incorrect answers.

Phase 4 Process Highlights:

  • First I mapped through the quizData that had been destructured into the QuestionsComponent which was established in App.jsx and passed down through the QuizPage component.
  • Then I Implemented an array to hold the correct answer object and used a spread operator to take in the 3 incorrect answers objects so that a single array could be pointed to for applying a shuffle function.
  • I turned to ChatGPT to find an algorithm for shuffling array elements called The Fisher-Yates Shuffle and used it to output the array of answers into random order.
  • I applied the map method to loop through the array and output each answer for each question to an HTML template
  • At this point I wanted to apply styles to indicate when a user had selected an answer and to allow them to change their minds and shift the style to the new answer if they so desired.
  • The final step in this phase of the project was to use the He library to decode HTML entities rendered in the data output.

The fifth phase involved processing the users answers, calculating the score and applying the styling to indicate correct and incorrect answer attempts after the answers have been submitted.

Phase 5 Process Highlights:

  • Inside the answer options that were mapped out using the map method an inline onClick handler in the rendered output in the QuestionsComponent was implemented to check for correct answers and applied green styling. Then for any incorrect answers it applied a pinkish red style and also a green style to indicate what the correct answer was.
  • When the user clicks Check Answers the setShowScore state sets showScore to true which activates the calculateScore function and passes the score to the CalcScore component for rendering.

The sixth phase was to repopulate the the QuizPage with new questions and answers data when the user clicks the play again button.

Phase 6 Process Highlights:

  • When the play again button is clicked the program fires the handlePlayAgain functon which clears the previous quiz data and sets the playAgainClicked state to true.
  • A useEffect that is initiated whenever the playAgainClicked state in the dependency is updated activates the fetch call to the OTDB using async await.
  • The handlePlayAgain function also sets the quizData state to null. This fires another useEffect which resets the score, clears the previously selected answers and sets answersChecked to false. All of which basically sets up the quiz page as a blank slate for the new questions and answers to be renderred again.

Lessons learned

Quizzical really helped me to understand the importance of controlling the calls to a fetch API when you do so using useEffect. This project also taught me that using ChatGPT to find solutions for project challenges is a game changer and saves so much time compared to scouring stack overflow and Google for potential solution. At the same time, I learned that I must have a good understanding of the web development language you're working with to fully take advantage of ChatGPT because sometime the AI misunderstands your intentions and you need to be able to spot the discrepencies and guide ChatGPT back onto the correct path. In the end I had a great understanding of how my code was working.
